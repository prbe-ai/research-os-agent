# os-agent v0.4 fold-in — Phase 2 spec

**Status:** Implemented (branch `feat/v04-foldin-phase2`, stacked on Phase 1). Depends on research-os PR #15 (`RunPatch` env_ref + foreign_keys).
**Context:** Phase 1 (`2026-07-11-v0.4-foldin-client-upgrade-plan.md`) wired the additive fold-in capabilities but left three items that needed either the `RunPatch` backend change or a version-download path. This is those three.

---

## 1. Interactive `foreign_keys` / `env_ref` (needs PR #15)

**Before (Phase 1 interim):** `link()` and `snapshot()` on the interactive path wrote `metadata.foreign_keys` / `metadata.env_ref`, because `RunPatch` couldn't set the real columns.

**Now:** research-os PR #15 adds `env_ref` (replace) + `foreign_keys` (per-key new-wins merge) to `RunPatch`.
- `run.link(**fk)` → `PATCH /v1/runs/{id} {foreign_keys: {...}}`. The server merges per-key new-wins, so a late-discovered id attaches without clobbering earlier keys and without a client read-modify-write round-trip.
- `run.snapshot()` → posts the content-addressed execution record, then `PATCH /v1/runs/{id} {env_ref: <content_hash>}` to pin the real `runs.env_ref` (composite FK to the execution record just created).
- `check_run()` reads the real `run.env_ref` (with a `metadata.env_ref` fallback for older runs).

Metadata interim removed. `run.foreign_keys` / `run.short_id` surface on reads (RunDetailOut).

## 2. Asset `materialize` (uses the existing download endpoint)

`client.assets.materialize(name, dest, *, kind=None, requirement=None)`:
1. `resolve(name)` → pick the selected version (latest, or by `requirement`);
2. read its `source_artifact_id` (the artifact it was zero-copy pinned from);
3. `POST /v1/artifacts/{source_artifact_id}/download` → presigned GET URL;
4. GET the URL (`transport.get_url`), write bytes to `dest`.

No new backend endpoint: reuses the fold #16 artifact download. A version pinned by `content_hash` only (no `source_artifact_id`) has no downloadable object and raises a clear error. CLI: `exp asset materialize NAME --to PATH`.

## 3. Upload `kind` / `meta` — deferred (needs a backend change)

`log_artifact(path=...)` uploads bytes via presign, but the backend `UploadRequest` carries only `name/content_hash/size/content_type/span/step` — no `kind`/`meta`, and there is no artifact PATCH. So an uploaded artifact is always `kind="file"` with no meta; the client warns once when `kind`/`meta` are passed with a `path`.

**This one cannot be done client-side.** It needs a small research-os change: add `kind` + `meta` to `UploadRequest` and the presign INSERT (or an artifact PATCH). Left as a follow-up backend PR; not in scope for the PR #15-based client work.

## Verification

- 43 client tests (mocked + real-git): `link` writes the real `foreign_keys` column with merge; `snapshot` pins the real `env_ref`; `materialize` downloads bytes (and errors on a content-hash-only version).
- Black-box against **research-os with PR #15**: the real sync `ros` client drives `run → log(dims) → link → snapshot → asset register/version` over HTTP against a live uvicorn instance on real Postgres, and reads back the real `foreign_keys` / `env_ref` columns. See `research-os` `tests/integration/test_ros_client_black_box.py`.
