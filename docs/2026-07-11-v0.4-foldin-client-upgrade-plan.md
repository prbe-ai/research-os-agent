# os-agent upgrade plan for research-os v0.4.0.0 (ingestion fold-in)

**Status:** Phase 1 IMPLEMENTED (40 tests green + cross-repo model validation). Decisions locked: promotion dropped; assets registry-only; events split into read (`events`) + note write (`notes`); `experiment_version()` added; FK/env_ref `RunPatch` parity deferred to Phase 2.
**Backend:** research-os `v0.4.0.0` (PR #13, merged), migrations 0004–0015. Reference: `research-os/docs/INGESTION_FOLDIN.md`, `docs/DESIGN_DIVERGENCES.md`.

---

## 1. What changed in research-os

PR #13 folded the 2026-07-09 core-ingestion design's primitives into the backend. All **additive**, and the `/ingest/v1/runs` response stays **byte-identical** (new server-stamped fields do not enter the frozen `RunOut`; they surface only on `/v1` reads via a new `RunDetailOut`). So the client's existing ingest keeps working; this is an upgrade, not a break.

New tables + surfaces:

| Fold | Backend surface | Closes which client workaround |
|---|---|---|
| #9 Metric dimensions | `metrics[].dimensions` (≤8 keys) on ingest and `POST /v1/runs/{id}/metrics`; series id widened to `(run,kind,key,dims_hash)` | `log_hw()` encoding dims into the metric key |
| #16 Presign upload | `POST /v1/runs/{id}/artifacts/uploads` → PUT to R2 → `POST /v1/artifacts/{id}/confirm`; `POST /v1/artifacts/{id}/download` | `log_artifact(path=)` recording a reference only, no bytes uploaded |
| #7 Execution records | `POST /v1/execution-records {code,deps,hardware,settings,paths}` (content-addressed); ingest top-level `execution_record` pins `run.env_ref` | `snapshot()` stuffing capture into `metadata.snapshot` |
| #5 Asset registry | `POST/GET /v1/assets`, `POST/GET /v1/assets/{id}/versions` (zero-copy pin from artifact) | `assets.*` failing with `CapabilityUnavailable` |
| #6 Experiment versions | `POST/GET /v1/experiments/{id}/versions` (immutable launch-time manifest) | `promote()` (no manifest existed) |
| #2 Lineage edges | `POST /v1/edges {source,relation,target}` (closed vocab), `GET /v1/runs/{id}/edges` | no lineage-edge write existed |
| #10 Events | `GET /v1/events`, `GET /v1/runs/{id}/events` (append-only, **server-emitted, read-only**) | client had no read; `events.add()` writes are unrelated (see D7) |
| #8 foreign_keys | `run.foreign_keys` on **ingest** (per-key new-wins merge); `GET /v1/runs?foreign_key=k:v` lookup | `link()` writing `metadata.foreign_keys` |
| #21 Run petnames | `runs.short_id`; `GET /v1/runs/{ref}` accepts UUID or petname | runs addressable only by UUID |
| #22 Project artifacts | `POST/GET /v1/projects/{id}/artifacts` | (new capability) |
| #13 Trajectory | convention: a `.jsonl` artifact `kind="trajectory"` (doc-only, no table) | formalizes trajectory capture |

**Rejected / not built (important):** `docs/DESIGN_DIVERGENCES.md` records that **promotion_tier (scratch/candidate/official) was NOT built**, the **arbitrary-depth spine was rejected** (top stays fixed `project → experiment → run`), and **IngestEnvelope was rejected** (replaced by run `short_id`). This directly invalidates parts of the client's aspirational asset/promote surface.

**Two real gaps for the agent-native (`/v1`) path:** `foreign_keys` and `env_ref` are only settable via **ingest** (`RunPatch` accepts neither). A run created with `create_run()` + logged interactively has no `/v1` way to set them today.

## 2. Client delta, by area (each with a decision)

### D1 — Metric dimensions (log_hw) · low risk, no backend gap
Today `log_hw(gpu_temp=88, device=3)` encodes `gpu_temp{device=3}` into the key. Change to pass the real `dimensions` map; also expose `dimensions=` on `log()`.
**Recommend:** do it. Drop the `_hw_key` encoder. `log_hw()` becomes `log(..., kind="hardware", dimensions={...})`.

### D2 — foreign_keys (link) · needs a decision
Real `runs.foreign_keys` exists but is **ingest-only**; `RunPatch` can't set it. Agent-native runs can't get real foreign_keys today.
**Options:** (a) keep `link()` writing `metadata.foreign_keys` as interim + document; (b) propose a tiny backend follow-up: accept `foreign_keys` on `RunPatch` (per-key new-wins merge, same as ingest); (c) only support foreign_keys on the ingest path.
**Recommend (b):** propose the backend `RunPatch.foreign_keys` merge (small, symmetric with ingest), and in the interim keep `metadata.foreign_keys` with a deprecation note. Also wire `foreign_keys=` through `client.ingest()` now (that path already works).

### D3 — Artifact presign upload (log_artifact path=) · high value, no backend gap
Implement the real 3-step flow: `POST .../artifacts/uploads {name, content_hash, size, content_type}` → `PUT` bytes to the returned presigned URL → `POST /v1/artifacts/{id}/confirm`. Fall back to reference/`uri=` as today when no bytes.
**Recommend:** do it. Adds one httpx `PUT` to R2 (no auth header on the presigned URL). Keeps fail-open semantics (spool the confirm).

### D4 — Execution records (snapshot) · needs a decision (same shape as D2)
Change `snapshot()` to `POST /v1/execution-records {code,deps,hardware,settings,paths}` (content-addressed, idempotent) instead of `metadata.snapshot`. The `git`/`env`/`gpu` capture maps to `code`/`deps`/`hardware`.
Gap: pinning `run.env_ref` to that record is **ingest-only** (`RunPatch` has no `env_ref`). Same fork as D2.
**Recommend:** post the execution-record now (content-addressed value regardless), record its `content_hash` on the run (interim `metadata.env_ref` + the D2 backend follow-up to accept `env_ref` on `RunPatch`), and set `execution_record` on `client.ingest()` so the ingest path pins `env_ref` properly.

### D5 — Asset registry (assets module rewrite) · needs a scope decision
The backend shipped `assets` + `asset_versions` only. There is **no** `resolve`/`materialize`/`fork`/`propose`/`promote` endpoint; "promote" == creating a zero-copy version from an artifact.
**Recommend:** rewrite `AssetClient` to the real surface — `register(name, kind)` → `POST /v1/assets`; `add_version(asset_id, from_artifact_id|content_hash, label)` → `POST /v1/assets/{id}/versions`; `list()/get()/versions()` reads. **Drop** `fork`/`propose`/`promote`-candidate (no backend; promotion_tier rejected). **Defer** `materialize` (needs an asset-version download path the backend doesn't expose yet — flag as a possible follow-up). Remove the `CapabilityUnavailable('asset_registry')` guard.

### D6 — promote() → experiment versions · needs a decision
`promote(run, approval, asset_refs)` assumed run-level promotion, which doesn't exist. The real manifest is `POST /v1/experiments/{id}/versions {label?, as_of?, exclude_run_ids?}` (an immutable snapshot of the experiment's runs).
**Recommend:** replace `client.promote()` with `client.experiment_version(experiment_id, label=...)` (create) + reads. Drop the run-level `promote` and the `--approval`/`--asset` CLI verb, or keep `exp promote` as an alias that creates an experiment version. Surface this rename for your call.

### D7 — events (naming collision) · low risk
Backend `events` = server-emitted lifecycle/structure log, **read-only**. The client's `events.add()` actually writes *research notes as artifacts* (intent/decision/observation) — unrelated to the backend table.
**Recommend:** add read methods `client.events.list()` / `run_events()` for the backend log; keep the research-note write but rename it to avoid the collision (`client.notes.add()` / `exp note add`), documented as "structured knowledge stored as a `kind=note` artifact." Small rename, clearer semantics.

### D8 — Lineage edges (new) · low risk
Add `client.add_edge(source_ref, relation, target_ref)` and `run.edges()` read. Enables train→eval and produces/consumes links (closed vocab from the backend enums).
**Recommend:** add a thin write + read. Model the closed vocab as enums generated from the schema.

### D9 — Run reads / short_id · low risk
`create_run`, `get_run`, `list_runs` now return `RunDetailOut` (adds `short_id`, `created_by`, `env_ref`, `foreign_keys`). `GET /v1/runs/{ref}` accepts a petname.
**Recommend:** expose `Run.short_id`, accept a petname in `get_run`/CLI `run` args, surface `foreign_keys`/`created_by` on reads.

### D10 — ingest() passthrough · low risk
Add `execution_record=`, per-metric `dimensions`, run `foreign_keys`, and `artifacts[].external_span_type` to `client.ingest()` (all now in `IngestRunRequest`). This is the one path where `foreign_keys`/`env_ref` actually pin server-side.
**Recommend:** do it.

## 4. Recommended phasing

**Phase 1 (no backend changes, high value):** D1 dimensions, D3 upload, D5 asset registry, D8 edges, D9 short_id/RunDetailOut, D10 ingest passthrough, D7 events read + note rename, D4 execution-record POST (record `content_hash` in metadata interim).

**Phase 2 (after the D2/D4 backend follow-up merges):** switch `link()` and `snapshot()` env_ref to real `RunPatch` fields; remove interim `metadata` fallbacks; reshape `promote` (D6); revisit asset `materialize`.
